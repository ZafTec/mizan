## 1. Workout Flow: Why It Is Broken + Rebuild Spec

The POST payload contract between frontend and backend actually aligns. What is broken is everything around it.

### 1.1 Blockers (the flow is dead without these)

**B1. The exercise catalog has ZERO seed data.**
Verified: no `HasData` for `Exercise` in `MizanDbContext.cs` (only achievements are seeded at line 672), no `InsertData` in any migration, no SQL seed scripts. Fresh install: library shows "0 exercises found", search returns nothing, the form requires >= 1 exercise (`workouts/page.tsx:210`), so **logging a workout is impossible**.
Fix: EF seed migration with ~80-150 exercises (Strength/Cardio/Flexibility/Balance, muscleGroup, equipment). Category casing must match the frontend literals `"Strength" | "Cardio" | "Flexibility" | "Balance"` (`page.tsx:24-30`).

**B2. No UI path to create a custom exercise.**
`frontend/data/exercise.ts:47` `createExercise` has zero call sites; admin exercises page is read-only. Backend `POST /api/Exercises` exists.
Fix: "Can't find it? Create exercise" inline form in the exercise picker.

### 1.2 Data loss and mangling

| ID | Bug | Evidence | Fix |
|---|---|---|---|
| D1 | One set value cloned N times; pyramids/drop sets unrepresentable, volume stats are fiction | `page.tsx:225-232` vs per-set `ExerciseSet.cs` model | Per-set rows in the form (spec 1.5) |
| D2 | Per-exercise notes destroyed into one semicolon-joined string | `page.tsx:224`; root cause: `WorkoutExercise` has NO Notes column | Add `Notes varchar(500)` to entity + DTOs + migration + codegen |
| D3 | Cardio duration multiplied by sets: "30 min run, 3 sets" stores 90 minutes | `page.tsx:33` + `:225-231` | Cardio sets = intervals with own duration, or hide Sets for cardio |
| D4 | `reps: 1` garbage stored on every cardio/flexibility set, history renders "1 reps" | defaults at `page.tsx:33-34`, render `:552` | Send `reps: null` for non-strength |
| D5 | `distanceMeters` exists in entity, DTOs, and history renderer but has NO input field | `ExerciseSet.cs:11`, render `page.tsx:557` | Distance input on cardio set rows |
| D6 | Default date is UTC: evening workouts get the wrong day, corrupts streaks | `page.tsx:75` `toISOString()` | Local date (`toLocaleDateString('en-CA')`) |
| D7 | Explicit 0kg (bodyweight) coerced to null by `\|\| null` idiom | `page.tsx:229` | `?? null` with empty-string checks |

### 1.3 UX wedges

- **U1. Gamification response discarded.** `page.tsx:217-234` ignores `LogWorkoutResult { streak, unlockedAchievements }` (generated type exists at `api.generated.ts:3557-3567`). The meals flow already does this right (`meals/add/page.tsx:26-44` + `GamificationToaster`). ~15 line fix, do it first.
- **U2. Library "Add" kicks you out of the library** (`page.tsx:192-195`), making a 6-exercise workout take 6 tab round-trips, even though a sticky "Back to Workout (N added)" button exists (`:902-913`). Fix: stay in library, toast.
- **U3. The in-progress workout lives only in client state.** Any refresh, navigation, or the 15-min JWT expiring mid-session (hard redirect in `api.client.ts:70`) wipes an hour of gym logging. Fix: persist draft to localStorage, offer "Resume draft?" on mount.
- **U4/U5.** `key={index}` on the editable list (`:703`) misassociates row state on remove; duplicate-add rules differ between search results and library. Fix: client `uid` per row (also legitimately allows two slots of squats).
- **U6.** Empty search results render nothing at all (`:672`): user cannot tell "no match" from "broken".
- **U7.** Frontend demands name + duration that the backend treats as optional (`:210` vs `LogWorkoutCommand.cs:81`).
- **U8. Unbounded numerics.** No min/max anywhere; `sets=100000` builds a 100k-element POST (backend has no count cap either); negative sets produce `Array.from({length:-3})` = `[]` = a stored exercise with zero sets. Fix both layers: UI `min=1 max=20`; backend validator: Exercises NotEmpty, <= 30 exercises, 1-50 sets, reps 1-1000, weight 0-1000.
- **U9.** `{workout.caloriesBurned && ...}` renders a literal "0" (`:526`). Use `!= null`.
- **U10.** "Stats" cards aggregate only the visible page of 10 (`:420-445`), numbers change when paging. There is no backend stats endpoint at all.
- **Entry point:** "Start Workout" on the dashboard lands on the **History** tab, not the log form (`page.tsx:71`, no `?tab=` support).

### 1.4 Backend gaps

- **G1. No GET-by-id, UPDATE, or DELETE for workouts** (`WorkoutsController.cs` has exactly 2 actions). A typo'd 800kg bench is permanent. Also `CreatedAtAction` fabricates a Location header for a route that does not exist (`:51`).
- **G2.** Unvalidated `exerciseId` hits the FK and 500s (`LogWorkoutCommand.cs:90-119`). Pre-validate, return 400.
- **G3.** `exercises: []` is accepted and still ticks streaks + `workouts_logged` achievements: 50 empty POSTs farm "Iron Habit". Fix: `RuleFor(Exercises).NotEmpty()`.
- **G4.** Category is free text matched case-sensitively everywhere (`GetExercisesQuery.cs:88`, `Exercise.cs:8` comment even says lowercase). A lowercase `"cardio"` exercise gets the strength form. Fix: enum-constrain + normalize + case-insensitive filter.
- **G5.** Trainer `CanViewWorkouts` permission is plumbed everywhere but no `GetClientWorkoutsQuery` exists: the toggle controls nothing.
- **G6-G8.** Wasted facet queries nobody consumes (`GetExercisesQuery.cs:125-128`, could instead REPLACE the hardcoded dropdowns and fix G4 drift); dead Includes (`GetWorkoutsQuery.cs:78-83`); MCP `list_exercises` sends `&search=` but the query binds `SearchTerm` (silently ignored, `ExerciseTools.cs:25`).

### 1.5 Rebuild spec

```
app/(dashboard)/workouts/
  page.tsx                  // SERVER: reads ?tab=&page=, fetches first history page
  WorkoutTabs.tsx           // client: tab shell, syncs ?tab= (QuickActions -> /workouts?tab=log)
  history/
    WorkoutHistoryList.tsx  // client: pagination, expand, receives server data
    WorkoutCard.tsx         // presentational; delete/edit actions
    WorkoutStats.tsx        // consumes new GET /api/Workouts/stats
  log/
    WorkoutLogForm.tsx      // client: useReducer machine, localStorage draft
    ExercisePicker.tsx      // debounced typeahead + category chips + inline create-exercise
    ExerciseEntryCard.tsx   // per-set rows, per-exercise notes, remove, reorder
    SetRow.tsx              // strength: reps+weight | cardio: duration+distance | flex: duration
                            // "duplicate last set" button = the 3x10 fast path
```

Form state: single `useReducer` over `DraftWorkout { name, date(local), durationMinutes, caloriesBurned, notes, exercises: DraftExercise[] }` where `DraftExercise { uid, exercise, notes, sets: DraftSet[] }` and `DraftSet { uid, reps, weightKg, durationSeconds, distanceMeters }`. Actions: ADD/REMOVE/MOVE_EXERCISE, ADD_SET (copies previous set values), UPDATE_SET, REMOVE_SET, UPDATE_EXERCISE_NOTES, UPDATE_META, RESET, HYDRATE_FROM_DRAFT. Persist to `localStorage["workout-draft"]`, clear on success.

Corrected POST (per-set, per-exercise notes after the Notes column lands):
```json
{ "name": "Push Day", "workoutDate": "2026-06-12", "durationMinutes": 62,
  "exercises": [
    { "exerciseId": "<guid>", "notes": "paused reps",
      "sets": [ { "reps": 10, "weightKg": 80 }, { "reps": 8, "weightKg": 85 }, { "reps": 6, "weightKg": 90 } ] },
    { "exerciseId": "<guid-cardio>",
      "sets": [ { "durationSeconds": 1200, "distanceMeters": 3000 } ] } ] }
```
Submit reads `LogWorkoutResult` into `GamificationToaster`, resets, clears draft, switches to history.

Backend work list: notes column migration; exercise seed migration; validator bounds; exerciseId pre-validation; `GET/PUT/DELETE /api/Workouts/{id}` (ownership-scoped); `GET /api/Workouts/stats?days=30` (total workouts, volume kg, minutes, per-muscle-group sets, max-weight PRs per exercise); category enum hardening; then `bun run codegen`.

### 1.6 Priority order

- **P0 (flow is dead without):** seed migration (B1) · read POST response into GamificationToaster (U1) · local-date default (D6) · `?tab=log` deep link + QuickActions target · backend Exercises NotEmpty + id validation (G2/G3).
- **P1 (quick correctness inside current component):** reps null for non-strength + stop multiplying cardio duration (D3/D4) · input bounds (U8) · stable uid keys (U4/U5) · `!= null` renders (U9) · no-results state (U6) · stay-in-library (U2) · delete dead/wrong `data/workout.ts logWorkout` (sends `date`/`sets:number`, would 400) · type the `any[]`s from generated types.
- **P2 (rebuild):** per-set UI + notes column + draft persistence + server/client split · edit/delete endpoints + UI · stats endpoint · custom exercise creation · category enums.
- **P3:** facets-driven dropdowns, MCP searchTerm fix, trainer workouts endpoint, dead Includes.

## 2. Feature-by-Feature Breakdowns and Fixes

### 2.1 Shopping Lists (BROKEN, highest-leverage fix in the app)

1. **The page crashes for anyone with data.** `meal-plan/shopping-list/page.tsx:39` renders `list.items`, but the list endpoint returns `ShoppingListSummaryDto` with NO items array, only counts (`GetShoppingListsQuery.cs:17-22,70-71`). `flatMap(l => l.items)` -> `[undefined]` -> `item.isChecked` -> TypeError.
2. **No create/add-item UI exists at all.** Backend `POST /api/ShoppingLists` and `POST /{id}/items` exist (`ShoppingListsController.cs:43,57`); the web UI has only a toggle, for items that cannot exist.
3. **"Generate from meal plan" does not exist at any layer.** The empty-state copy ("Add meals to your meal plan to generate a shopping list", line 135) is false.
4. `DeleteShoppingListCommand` and `UpdateShoppingListItemCommand` exist in Application with **no controller routes**: unreachable dead handlers.

Fixes, in order: (a) include items in the list query or fetch `GET /api/ShoppingLists/{id}` per list; (b) New List + Add Item forms (endpoints exist); (c) new `GenerateShoppingListFromMealPlanCommand`: walk plan recipes -> `RecipeIngredients` -> aggregate by `FoodId` + unit, scale by servings; idempotent regeneration (replace unchecked auto items, keep manual/checked); route `POST /api/MealPlans/{id}/shopping-list`; (d) wire the two dead commands to routes.

### 2.2 Meal Plans (PARTIAL)

- Create page is hard-locked to the current week: `const [startDate] = useState(getMonday(new Date()))`, no setter (`meal-plan/create/page.tsx:51`). UTC `formatDateKey` shifts slots for users west of UTC (`:40-42`).
- `/meal-plan/add` spawns a new one-day plan named "Meal Plan" per submission (`add/page.tsx:115-128`) and redirects with a `?week=` param the list page never reads.
- **`components/MealPlanningCalendar/index.tsx`, the weekly calendar with logged-vs-planned comparison and week navigation, is imported by NOTHING** (still uses MongoDB-era `_id` fields). It is the missing unifier.
- List page "This Week" stat is actually "plans with any recipes" (`meal-plan/page.tsx:94`).
- Edit's "Add Recipe" always inserts at startDate + dinner (`MealPlanEditForm.tsx:79-86`).
- Connective tissue absent: no "log planned meal to diary" (no command, no UI), dashboard never shows today's planned meals, frontend never sends `householdId` though the backend accepts it.

Fixes: week picker on create + kill or repoint `/add` to "add into existing plan covering the date" + local date keys (quick). Structural: resurrect MealPlanningCalendar as the meal-plan home with a one-plan-per-week model; add "Log this slot" action that POSTs `/api/Meals` from a plan recipe; "duplicate to next week" button; ask date+mealType in the RecipeSearchModal.

### 2.3 Goals (PARTIAL)

- Nav "Goals" points at the **form**, not the dashboard; `/goal/dashboard` is only reachable via a quick-link; every save creates a NEW goal row and deactivates the old (`CreateUserGoalCommand.cs:82-108`), so tweaks pollute history; no end/edit/delete endpoints exist.
- `/goal/progress` demands manual re-entry of calories/protein/carbs/fat that the diary already knows; the page's own tip admits it ("Use the food diary to calculate totals automatically", line 87). `RecordGoalProgressCommand` never reads `FoodDiaryEntries`.
- Goal history endpoint exists and feeds chart reference lines, but no page lists it.
- Goal `actualWeight` and body-measurement `weightKg` are two unconnected weight stores.
- `goalType` select allows saving a typeless goal ("" option).
- Dashboard's "Today's Progress" shows the last manual entry, which may be days old.

Fixes: nav -> `/goal/dashboard` with "Set targets" secondary (quick); prefill progress from `GET /api/Nutrition/daily` (quick); derive GoalProgress from the diary server-side with manual override (structural); single weight source of truth with measurements (structural); goal history view + explicit end-goal.

### 2.4 Body Measurements (WORKS, one gap)

No edit: controller is GET/POST/DELETE only (`BodyMeasurementsController.cs`). A typo means deleting and re-entering 11 fields. Delta colors ignore `goalType` (weight up = always orange, wrong for muscle_gain users). Logging weight does not update goal progress. Fix: PUT endpoint + row edit (quick); goal-aware delta colors (quick).

### 2.5 Households (BROKEN as a feature)

Membership admin works end-to-end. But the page claims "Share recipes, meal plans, and shopping lists" and "New items are shared here by default" (`profile/household/page.tsx:13,46,62`) and **all three claims are false**:
- Recipes: `GetRecipesQuery.cs:84-86` is `owner || IsPublic`, no household clause anywhere.
- Meal plans: `GetMealPlansQuery.cs:57` filters own only. (Direct-ID household access exists in `GetMealPlanByIdQuery.cs:137-140`, for an ID the list never reveals.)
- Shopping lists: own only (`GetShoppingListsQuery.cs:54`).
- `ActiveHouseholdId` is written by `SetActiveHouseholdCommand` and **read by nothing**; the frontend never sends `householdId` on any create.
- Invites send no email and no notification; the invitee discovers them only by visiting `/profile/household`.
- `CanEditShoppingList` permission is set on join and enforced nowhere.

Fix (small surface, schema already has the columns): stamp `ActiveHouseholdId` onto created plans/lists server-side; extend the three list queries with `OR HouseholdId IN (my households)`; household recipe visibility; surface it (dashboard household card, "Household" recipes filter chip, shared badge on shopping lists). Or, until then, delete the sharing claims from the copy. See also §3 Spec 5 for the invite loop.

### 2.6 Trainers (BROKEN end-to-end)

Client side: discovery + request works, then dead ends:
- `/trainers/requests` and `/trainers/my-trainer` have **zero inbound links** anywhere (grep confirmed).
- Cancel request: toast "not available yet" (`requests/page.tsx:124`).
- "Send Message" links to `/chat?recipientId=...` which **does not exist** (real route: `/messaging`) -> 404 (`my-trainer/page.tsx:228`). Same broken link in `ClientHeader.tsx:117`.

Trainer side:
- **Becoming a trainer via UI is impossible.** Admin role dialog offers "Trainer" but the handler silently skips it (`admin/users/[id]/user-actions.tsx:75-99`: `if (role === "user" || role === "admin")` with a TODO) and still toasts success. Only direct DB edits create trainers.

- What works: pending requests accept/decline, client list, client nutrition view with permission-aware errors.

Fixes: fix the three `/chat` links to `/messaging` (5 min); link requests/my-trainer from `/trainers` and profile; wire admin trainer-role assignment for real (backend admin endpoint or BetterAuth role update); delete fake fallbacks, compute stats from `/api/Trainers/clients`; add cancel-request + end-relationship commands; client-side permission editing.

### 2.7 AI (chat PARTIAL, suggestions BROKEN)

- The plugin gives the model 5 real tools including **LogFood, which writes diary entries** (`NutritionPlugin.cs`), auto-invoked. The `/ai` sidebar says the assistant "can't log meals or update goals for you": the UI lies in the safe direction, hiding the killer feature and surprising users with silent writes.
- **No conversation memory:** each request builds a fresh history with only system + latest message (`NutritionAiService.cs:48-50`) while the UI shows a transcript. "Make the second one vegetarian" silently fails. Fix: accept `messages[]`.
- Errors swallowed into a friendly apology string with HTTP 200 (`:67-71`): a bad API key is indistinguishable from advice.
- Suggestions: `data/suggestion.ts:31-38` prompts the chat endpoint for JSON and regex-extracts it, but the `GetRecipeSuggestions` tool returns **titles only** (`NutritionPlugin.cs:182-185`), so the model **fabricates recipe IDs** and every card links to `/recipes/{hallucinated-id}` -> 404. Fixes: transcript support; honest capability copy; structured suggestions endpoint returning real recipe IDs (skip JSON-from-prose); "Log it" / "Add to plan" actions on suggestion cards; link /suggestions from dashboard + nav; surface AI errors as errors; wire analyze-image into the meal-log Photo tab (§3 Spec 1).

### 2.8 Achievements, Habits, Community, Notifications

- Achievements WORK (seeded criteria, progress bars toward locked badges already render at `achievements/page.tsx:198-208`). Gaps: `recipes_created` and `goal_progress_logged` criteria exist but their commands never call the evaluator (badges unlock late on unrelated triggers); the streak header never says what sustains it.

### 4.7 Conventions (paste into the repo)

```markdown
## SVG & icon conventions
### Icons
- ONE source: lucide-react via the shared Icon path. Never icon fonts, never
  hand-pasted <path> data, never per-page imports from other sets.
- Sizes: 16 (dense/inline), 20 (default UI), 24 (page headers). No other values.
- Stroke width 2. Outline grammar only, no filled glyphs.
- Color: currentColor only. Never hardcode hex on an icon.
- Decorative icons: aria-hidden="true" (wrapper defaults to it). Icon-only
  controls: the parent carries aria-label.
### Decorative / background art
- Prefer CSS (gradients, masks, color-mix, blur) over SVG for glows/grids/blobs.
- Always aria-hidden="true" + pointer-events-none.
- Colors from theme tokens (var(--color-*)) so light/dark both work.
- No feGaussianBlur/feDropShadow inside SVG; use CSS filter on the wrapper.
- Any animation respects @media (prefers-reduced-motion: reduce).
### Illustrations
- Inline React components in components/illustrations/, not public/ files.
- Brand hue families + neutrals only. Max ~4 KB optimized (SVGO, keep viewBox).
- No embedded <text>. Validate hex values (broken hex renders black).
### Charts (recharts)
- Series colors from var(--chart-1..4); macro semantics from the shared
  MACRO_COLORS map (protein=verdigris, carbs=tuscan-sun, fat=sandy-brown,
  calories=burnt-peach). Grid/axes/tooltips: var(--border)/var(--muted-foreground)/var(--popover).
```
