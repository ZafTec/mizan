## 1. Data Security

### CRITICAL/HIGH

**S1. Plaintext secrets in the audit log.** `Mizan.Application/Behaviors/AuditBehavior.cs:67` serializes every `*Command` request into `AuditLogs.Details`. `ValidateTokenCommand` carries the raw MCP bearer token, so every validation writes the live secret to the DB — defeating the SHA-256 `TokenHash` design. Chat message content is also persisted this way (privacy).
**Fix:** `ISkipAudit` marker interface or `[SensitiveData]` redaction before serialization; exclude `ValidateTokenCommand` and `SendChatMessageCommand`.

**S2. Service-key impersonation is global.** `Mizan.Api/Authentication/ApiKeyAuthenticationHandler.cs:51-83` + `Program.cs:101-117`: the static `Mcp:ServiceApiKey` is part of the DefaultPolicy, so every `[Authorize]` endpoint accepts it, and `X-Impersonate-User: <guid>` mints a principal with that user's real role — impersonating an admin yields admin. Impersonation skips the email-verified check that the JWT path enforces.
**Fix:** Dedicated MCP-only policy instead of folding ApiKey into DefaultPolicy; disallow impersonating admins; require email-verified; use `CryptographicOperations.FixedTimeEquals`; rotate the key.

**S3. Weak fallback secrets in docker-compose.** `docker-compose.yml:13,60,129,156`: `mizan_dev_password`, `dev_secret_change_in_production` (BetterAuth signing!), `dev_mcp_key_change_in_production` as defaults. Combined with S2, a deploy from this file is fully compromised. Dev compose also publishes Postgres (weak password) and Redis (**no password**) on the host (`:16-17`, `:30-31`).
**Fix:** Remove fallbacks (fail fast like `Program.cs:97-98` does); bind dev ports to 127.0.0.1; set a Redis password. Prod compose is already clean.

### MEDIUM

**S4. JWT issuer/audience validation silently self-disables.** `JwtAuthenticationExtensions.cs:41-44`: `ValidateIssuer = !string.IsNullOrWhiteSpace(jwt.Issuer)` — empty config = validation off, no error.
**Fix:** Make Issuer/Audience mandatory in `JwtOptions` validation; set `ValidateIssuer/Audience = true` unconditionally.

**S5. AI image upload unbounded.** `NutritionController.cs:60-74`: only checks `Length == 0`, buffers entire `IFormFile` to memory, forwards to OpenAI. Memory DoS + uncontrolled OpenAI spend.
**Fix:** `[RequestSizeLimit]` + content-type allowlist + max-byte check before buffering.

### LOW

- **S6.** `CreateExercise` is plain `[Authorize]` while Foods/Achievements creation is admin-only; custom exercises (with attacker-controlled `VideoUrl`/`ImageUrl`) appear in the shared catalog (`ExercisesController.cs:27-33`). Fix: admin-gate or scope `GetExercises` to global + own.
- **S7.** `/api/McpTokens/validate` is `[AllowAnonymous]`, unthrottled token oracle (`McpTokensController.cs:71-83`). Fix: IP rate limit.
- **S8.** `/api/Users/me/debug` dumps full claim set in prod (`UsersController.cs:42-81`). Fix: dev-only or delete.
- **S9.** `AddRecipeToMealPlanCommand.cs:49-89` doesn't verify recipe access — private recipe metadata leaks via meal-plan read. Fix: require public/owned/household recipe.
- **S10.** `ChatHub.TypingIndicator` (`ChatHub.cs:231`) broadcasts without re-checking membership (cosmetic).

### Verified clean
Ownership/IDOR checks consistent across all handlers (meals, recipes, measurements, goals, meal plans, shopping lists, households, trainer scopes). No mass assignment of `UserId`/`Role`. No raw SQL in app code. No committed secrets (`.env*` ignored, only placeholders in `.env.example`). XSS surface clean (`dangerouslySetInnerHTML` only on static JSON-LD/theme script). JWT held in memory, not localStorage. BetterAuth hardening solid (HIBP check, per-endpoint rate limits, httpOnly+lax+secure cookies). CORS origin-restricted. Note: CLAUDE.md says ES256 but implementation is EdDSA/Ed25519 — doc drift.

## 2. Anti-Patterns

### HIGH

**A1. JWKS failure cached** (top-10 #4). `JwksProvider.cs:60-72` catches all, returns `""`, which HybridCache caches → all auth fails until TTL expiry.
**Fix:** Throw (HybridCache won't cache failures) or stale-while-revalidate keeping last good key set.

**A2. Sync-over-async in the JWT validator hot path.** `EdDsaJwtSignatureValidator.cs:27`: `.GetAwaiter().GetResult()` on Redis+HTTP I/O per authenticated request on cache miss → thread-pool starvation under load.
**Fix:** Async key resolution via `JwtBearerOptions` events or a hosted-service pre-warmed in-memory snapshot.

**A3. Recipe nutrition math exists in 4 copies.** `CreateRecipeCommand.cs:171-309`, `UpdateRecipeCommand.cs:185-279`, `recipes/add/page.tsx:57-79`, `EditRecipeForm.tsx`. Copies already drift (fiber/ratio).
**Fix:** Pure `RecipeNutritionCalculator` in `Mizan.Domain`; frontend shares one `lib/nutrition.ts` or displays backend-computed preview.

**A4. EditRecipeForm zeroes ingredient macros.** `EditRecipeForm.tsx:59-78` hydrates existing ingredients with `caloriesPer100g: 0,...` → edited recipes preview wrong nutrition. Direct consequence of A3.
**Fix:** Recipe detail endpoint should return per-100g macros for food ingredients (it already does for sub-recipes — the asymmetry is the bug).

### MEDIUM

- **A5.** No `AsNoTracking()` on ~32 of 37 query handlers; worst is `GetRecipeByIdQuery.cs:59-68` which also has 4 collection Includes in one SQL query (cartesian explosion: 20 ingredients × 15 steps × 5 tags = 1,500 rows). Fix: `.AsNoTracking().AsSplitQuery()`.
- **A6.** Two competing error conventions: handlers returning `{Success=false, Message}` mapped to blanket 400s (`RecipesController.cs:51-64`) vs the well-built exception→status mapping in `Program.cs:253-317`. Clients can't tell 404 from 403. Fix: throw `EntityNotFoundException`/`ForbiddenAccessException`, delete Success-flag results.
- **A7.** `AuditBehavior.cs:45` does a second `SaveChangesAsync` on the handler's same scoped DbContext (flushes anything intentionally unsaved); `GetProperty("recipeId")` camelCase probes never match PascalCase props (dead code). Fix: separate context or queued background writer.
- **A8.** Authorization after expensive work in `UpdateRecipeCommand.cs:83-106` (recursive circular-dependency walk before the ownership check). Fix: check ownership first.
- **A9.** `RecipeCircularDependencyValidator.cs:35-44`: recursive N+1, one DB query per node, called once per sub-recipe ingredient, no CancellationToken. Fix: load edge list once, cycle-check in memory.
- **A10.** No pagination clamping: `QueryableExtensions.cs:8-13` trusts raw `Page`/`PageSize` — `?pageSize=1000000` dumps tables; `page=0` → negative Skip exception. Fix: `Math.Clamp(pageSize, 1, 100)`, `Math.Max(1, page)`.
- **A11.** Controllers never forward `CancellationToken` (all of `Mizan.Api/Controllers/`); abandoned requests run to completion. Fix: add `CancellationToken ct` to every action.
- **A12.** `GetProfileExportQuery.cs:220-424`: 12 sequential unbounded `ToListAsync` (incl. `McpUsageLogs`, all diary entries ever) in one user-triggerable request. Fix: cap/paginate or background job + streamed export.
- **A13.** Achievement rules (business policy, magic strings `"meals_logged"`) live in Infrastructure (`AchievementEvaluator.cs:139-150`) with 8 sequential count queries per evaluation. Fix: pure rules in Domain, single grouped stats query, skip irrelevant criteria per trigger.
- **A14.** `ChatHub.cs:41-47` injects `IMizanDbContext` directly into the presentation layer, duplicating authorization logic that `SendChatMessageCommand` also has. Fix: a `CanAccessConversationQuery` / reuse trainer auth service.
- **A15.** `workouts/page.tsx`: 918-line client god-component, 19 useState hooks, `any[]` API types despite codegen, `key={index}` on editable list, per-exercise notes flattened into one string (silent data loss at line 224). Fix: split into server-component history + client form island; use generated `ExerciseDto`/`WorkoutDto`.
- **A16.** All 15 `data/*.ts` files are `"use server"` actions used for READS (serialized POSTs, uncacheable; typeahead through an action queue). Fix: actions for mutations only; reads via `clientApi` GETs or plain server functions.
- **A17.** Recipe add page (742 lines) and EditRecipeForm (478 lines) are parallel implementations that already diverged: add page debounces search with race guard; edit form fires a request per keystroke with no debounce and no out-of-order protection (`EditRecipeForm.tsx:129-142`). Fix: one `RecipeForm` with `initialRecipe?` prop.

### LOW
Empty catch in `ValidateTokenCommand.cs:60-68`; redundant `Include`+`Select` in `GetRecipesQuery`/`GetFoodDiaryQuery`; AI failures returned as success-shaped values (`NutritionAiService.cs:67-70,139-142`); manual `new RecipeCircularDependencyValidator(_context)` instead of DI; magic numbers (token length 68, 30000ms skew, role string literals); `router.push` during render + `window.location.reload()` in `profile/settings/page.tsx:84-87,167`; queries returning empty 200 for missing users instead of 401.

### Notably clean
Thin controllers, consistent CQRS dispatch, Application has zero Infrastructure references, centralized pagination/sorting, disciplined `lib/api.ts` split, `dashboard/page.tsx` is a proper parallel-fetching server component.

## 3. Data Flow

**End-to-end as built:** every page is `force-dynamic`; server reads go page → `"use server"` data fn → `serverApi` → `getSession` + HTTP **self-call** to `/api/auth/token` → backend JWT validation (Redis JWKS) → MediatR → EF → Postgres, with `convertKeysToCamelCase` on the way back. Writes return through `useActionState` + `router.refresh()`. Almost no `revalidatePath`/`revalidateTag`.

### HIGH

**D1. Account deletion is broken** (top-10 #3). `lib/auth.ts:38-40` enables `deleteUser` with no cleanup hook; EF created 23 FKs to `users`. `household_invitations` FKs are `Restrict` (`MizanDbContext.cs:143-144`) → deletion **500s** for anyone who ever touched an invitation; elsewhere cascades silently destroy diaries/workouts/goals/chat while recipes go author-less via SetNull.
**Fix:** BetterAuth `user.delete.before` hook → backend transactional cleanup/anonymize command → then auth delete. Test the invitation case.

**D2. Meals page = 5 serialized server-action round-trips.** `meals/page.tsx:122-182` is a client page calling 4 server actions via `Promise.all` (Next executes actions **serially** per client) plus a 5th sequential `getNutritionRange`. Each action pays `getSession` + token self-fetch + backend hop. The app's most-used page.
**Fix:** Server component with genuinely parallel fetches (like the dashboard), or `clientApi` GETs.

**D3. Chat real-time is fiction** (top-10 #7). `MessagingThread.tsx:29,77-103` polls full conversation every 15s and refetches everything after send; `lib/signalr.ts` imported by nothing; backend REST send path never broadcasts (no `IHubContext` usage); `GetChatConversationQuery.cs:40-43` loads the entire unpaginated history every poll. If revived, `signalr.ts` has a stale-token `accessTokenFactory` (15-min JWT → reconnects fail) and no `onreconnected` group rejoin.
**Fix:** Either wire SignalR end-to-end (inject `IHubContext<ChatHub>` into the send command; `accessTokenFactory: () => getApiToken()`; rejoin groups + fetch-since on reconnect) or delete the hub and own polling — with `Take(N)` + `?since=` incremental fetch either way.

**D4. Profile update dual-writes the same row from two ORMs.** `profile/settings/page.tsx:147-159` runs `authClient.updateUser` AND `PUT /api/Users/me` via `Promise.all` — same physical `users` row, written by Drizzle and EF concurrently; contradicts `User.cs` "Backend CANNOT modify users table". Partial failure desyncs navbar (5-min session cookie cache) from data.
**Fix:** Single writer — `authClient.updateUser` alone suffices; delete `UpdateUserCommand` or make it the only path.

**D5. UTC "today" bugs.** `data/meal.ts:42,69` + `GetFoodDiaryQuery.cs` use UTC date; `meals/add/page.tsx:16-20` mixes local time-of-day with UTC date; `StreakService.cs:31` uses `DateTime.UtcNow`. A user in UTC-7 logging dinner writes into "tomorrow", dashboard misses evening meals, and streaks break randomly.
**Fix:** Pass the client's local date everywhere (diary page already does via `?date=`) or store profile timezone; feed it into `RecordActivityAsync`.

### MEDIUM

- **D6.** No backend transactions anywhere (grep: zero `BeginTransaction`). Meal log = entry save → streak save → achievement save; mid-chain failure leaves entry saved but returns 500 → user re-logs → duplicate (`CreateFoodDiaryEntryCommand.cs:122-126`). Fix: transaction (or MediatR transaction behavior), or idempotent queued side effects.
- **D7.** `revalidatePath("/settings/household")` targets a route that doesn't exist — real route is `/profile/household` (`data/household.ts:94-165`). Currently masked by force-dynamic. Fix the path or delete the calls.
- **D8.** `UserStatusService` 2-min cache is never invalidated, and bans happen via the frontend (Drizzle) so the backend never sees the write — banned users keep API access for up to ~2 min + JWT lifetime (`UserStatusService.cs:10-34`). Also: the `user == null` result is cached 2 min, so a request racing signup can lock the new user out briefly (`:44-45`). Fix: backend invalidation endpoint called after ban/role changes; don't cache negative lookups.
- **D9.** Dashboard double-fetches `GET /api/Meals?date=today` (`dashboard/page.tsx:48-53` calling both `getTodayMeal` and `getDailyTotals`). Fix: one `getFoodDiary(date)` returning `{entries, totals}`.
- **D10.** `serverApi` pays N session lookups + N internal HTTP hops per page (`api.server.ts:30-59`). Fix: React `cache()` per-request memo + in-process `auth.api.getToken()` instead of self-fetch.
- **D11.** Hand-written DTO interfaces shadow generated types across `data/` (`meal.ts:10-35` declares `foodId`/`recipeId` the backend never returns; `recipe.ts:15-31` Omit-patches for fields codegen already has). Concrete shipping bug: `meal-plan/add/page.tsx:46-52` reads `recipe.calories` (doesn't exist; it's `nutrition.caloriesPerServing`) → every recipe in the picker shows 0 kcal. Fix: alias `components["schemas"]` everywhere (the `types/api-contracts.ts` pattern).
- **D12.** Log-meal form collects `datetime-local` but `data/meal.ts:105-106` strips the time and never sends `loggedAt` (which the backend supports) → back-dated entries get `LoggedAt = UtcNow`, breaking diary ordering. Fix: send `loggedAt`.
- **D13.** Client-side filtering over truncated data: `meal-plan/add/page.tsx` fetches first 50 recipes then filters in-client (recipes 51+ unfindable, no truncation hint); same on `trainers/page.tsx:78`. Fix: server-side `SearchTerm` (RecipeSearchModal already does it right).
- **D14.** `meal-plan/add/page.tsx:115-128` hardcodes plan name "Meal Plan" and creates a NEW single-day plan per submission instead of adding to an existing plan covering the date — plan rows multiply.
- **D15.** Streak read-modify-write race, no concurrency token / unique index on `(UserId, StreakType)` (`StreakService.cs:33-77`). Fix: unique index + conflict retry or atomic UPDATE.
- **D16.** `cachedToken` not cleared on sign-out (`api.client.ts:7`) — safe today only via full-page navigation; any SPA logout reuses previous user's JWT up to 15 min. Fix: null it in sign-out paths.
- **D17.** Household invite concurrent-accept hits composite-PK violation → unhandled 500 (`RespondToHouseholdInvitationCommand.cs:49-117`). Catch and return "already accepted".

## 4. UI/UX

### HIGH

**U1. Silent failure → fake empty states** (top-10 #6). All 13 `data/*.ts` files `catch { return []/null }`; the meals page's error state is unreachable (`meals/page.tsx:173-178`). Outage = "No meals logged" = users think their data is gone. Error boundaries exist but are dead code for these flows.
**Fix:** Rethrow or return `{ok, data|error}`; keep catch-and-default only where missing is semantically valid (e.g. `getCurrentGoal` 404).

**U2. Missing loading.tsx + force-dynamic = frozen navigation.** Root layout forces dynamic (`app/layout.tsx:17`); loading.tsx missing for dashboard, achievements, ai, body-measurements, community, exercises, habits, messaging, goal/*, all of admin/*, public /recipes, and no root `app/loading.tsx`. Clicking nav gives zero feedback until all fetches resolve.
**Fix:** `loading.tsx` skeletons at `app/`, `app/(dashboard)/`, `app/admin/`.

**U3. /recipes lives outside the dashboard shell.** Sidebar links to public `/recipes` (marketing navbar, no sidebar/bell); "Create Recipe" jumps back into the shell. Layout thrash in the most-trafficked flow (`AppShell.tsx:44` vs `app/recipes/page.tsx`).
**Fix:** Authenticated recipes list inside `(dashboard)`; keep the public route for SEO/anon.

**U4. No modal traps focus or declares dialog semantics.** Zero `role="dialog"`/`aria-modal`/focus management across `Modal.tsx`, `ModalShell.tsx`, `DeleteConfirmModal.tsx`, `ConfirmationModal.tsx`, AppShell logout modal, RecipeSearchModal.
**Fix:** Radix Dialog (already shadcn-style) or add semantics + trap + focus restore to `ModalShell` and migrate.

**U5. Chat dead end + draft loss.** `MessagingThread.tsx`: input AND send disabled when no conversation exists while the empty state says "Say hello" (first message can never be sent unless backend pre-creates conversations); `setInput("")` runs before the POST so a failed send destroys the draft (line 96).
**Fix:** Create-conversation-on-first-send; restore draft on failure.

**U6. Fake global search.** `AppShell.tsx:405-414`: the prominent header input "Search foods, recipes, workouts…" has no handler at all.
**Fix:** Wire to foods/recipes search or remove until built.

### MEDIUM

- **U7.** Goal form clobbers user input when the async load resolves; `loadingGoal` computed but never rendered (`goal/page.tsx:47-67`). Also `target!` non-null assertions → `Infinity%` when targets missing (`goal/dashboard/page.tsx:86-89`).
- **U8.** Silent mutation failures: meal delete has no else-branch (`meals/page.tsx:108-120`), shopping-list toggle catch-logs only. Fix: error toasts.
- **U9.** `window.confirm` in household/admin destructive flows vs styled `ConfirmationModal` elsewhere — pick one (the modal).
- **U10.** "Add to Meal Plan" button actually logs to the diary (`RecipeActions.tsx:71-92`). Rename + add a real plan action.
- **U11.** Recipe add: no unsaved-changes guard anywhere in the codebase (back arrow discards a fully built recipe); submit doesn't require ≥1 ingredient (`amount: ing.amount!`); single shared `dropdownRef`/`ingredientSearch` across mapped rows breaks click-outside + shows stale results (`recipes/add/page.tsx:48-53,215-254,457`).
- **U12.** Raw error text shown to users: `error.tsx` renders `error.message` in a monospace box; `lib/toast.ts:20` surfaces unparsed `raw` body (can be an HTML error page); chat shows "API error: 500".
- **U13.** Meal-plan create is locked to the current week — no week picker (`meal-plan/create/page.tsx:51`); `toISOString()` date keys shift week boundaries west of UTC.
- **U14.** `AddMealFromRecipe` never submits the servings value — no hidden input, so backend defaults to 1 even when user logged 2× (`components/AddMealFromRecipe/index.tsx:32` vs `data/meal.ts:104`). One-line fix.
- **U15.** Icon-only buttons missing aria-labels throughout (day chevrons, back arrows, +/- steppers, tag/recipe remove X — the hover-only `opacity-0 group-hover` delete is invisible to keyboard/touch). ~31 aria-labels exist app-wide.
- **U16.** Social sign-in has no try/catch (stuck spinner on failure); register has no name field — users get greeted by their email prefix (`login/page.tsx:89-98`, `register/page.tsx`).
- **U17.** Empty-state copy duplicated ("No meals logged" twice) and filtered-empty shows "Be the first to add a recipe!" instead of "no results, clear filters".

### LOW
Notification bell has no unread badge (field exists, never populated); diary navigates into the infinite future; charts render blank axes for new users with no CTA overlay; recipes list double-fetches for a favorites-count vanity stat; `DeleteConfirmModal` wedges if `onConfirm` throws (no finally); pagination renders focusable disabled links; marketing navbar and AppShell duplicate logout/avatar logic wholesale; mobile bottom nav lacks a quick-log FAB for the highest-frequency action.

### Calibration: what's good

## 5. User Engagement Blockers

### CRITICAL

**E1. The core loop is the most expensive version of itself.** `meals/add/page.tsx:56-199` is a manual form: name + meal type + 5 hand-typed macro numbers. No food search, no recents, no favorites, no copy-yesterday, no barcode. Meanwhile the backend already has `POST /api/Nutrition/log` (logs by foodId, runs streaks/achievements) and `POST /api/Nutrition/ai/analyze-image` (photo logging — advertised as a Pro feature on the landing page) — **neither has any UI**. The foods catalog at `/ingredients` is a read-only table with no "log this" button, and food creation is admin-only.
**Fix (highest ROI in this report):** Search-first log flow on the foods DB (Redis-cached search endpoint exists), quantity stepper, recents tab derived from diary history, "duplicate yesterday", photo upload wired to analyze-image. Effort L (M for just wiring existing endpoints).

**E2. Signup gauntlet.** register → "check your email" → verify → land on `/login` → re-type credentials → `/` → `/dashboard`. Two authentications + an inbox context-switch before any value.
**Fix:** BetterAuth `autoSignInAfterVerification` + land verified users on `/dashboard?welcome=1`. Effort S.

**E3. Zero re-engagement channels.** `/notifications` is a hardcoded placeholder ("v2 rollout"); the bell has no badge; no reminder emails (nodemailer is configured but only sends verification/reset); CLAUDE.md's NotificationHub/GoalHub don't exist (only ChatHub is mapped). Nothing ever brings a user back.
**Fix:** Real notifications table fed by invites/trainer requests/achievements + bell badge (S/M); streak-at-risk evening email via existing nodemailer + cron (M).

**E4. No funnel analytics.** GA4 captures page_views only — zero `sendGAEvent` calls. `sign_up`, `meal_logged`, `goal_created`, `streak_broken`, invite events: none instrumented. You cannot measure any fix in this report.
**Fix:** ~10 `sendGAEvent` calls at existing success callbacks. Effort S.

### HIGH

- **E5. Streaks are fragile and randomly unfair.** UTC boundary (see D5) breaks streaks for evening loggers west of UTC; one missed day resets a 100-day streak with no protection (`StreakService.cs:31,67`). Fix: local-date streaks + 1 grace day or earnable freeze. Effort M.
- **E6. The habits checklist is unwinnable.** Habit page checks for `breakfast/lunch/dinner` meal types (`habits/page.tsx:29-60`) but every logging UI only offers `MEAL | SNACK | DRINK` — 3 of 5 daily habits can never be completed; permanent failure state. Fix: add Breakfast/Lunch/Dinner/Snack types, default by time of day. Effort S.
- **E7. Household invites have ~zero viral coefficient.** Invitee must already be registered (`InviteHouseholdMemberCommand.cs:59-62`), no email is ever sent, and the invite inbox is buried at Profile → Household. Five out-of-band coordination steps. Fix: tokenized invite email that survives registration and auto-accepts post-signup; surface pending invites on the dashboard. Effort M.
- **E8. Cold start is a blank slate.** No seeded foods or recipes (only achievements are seeded); non-admins can't create foods; empty states offer no path. Fix: seed ~200 USDA staples + 10-20 starter recipes as system content. Effort M.
- **E9. No onboarding.** New user lands on zeroed rings; only nudge is a small "Set a goal" button leading to a ~12-field form. Fix: 3-step wizard (goal type → TDEE → accept macros, reusing the existing TDEE calculator) + dismissible dashboard checklist. Effort M.

### MEDIUM

- **E10.** Achievement unlocks are silent outside the meal-add flow — `GamificationToaster` only mounted there; workout/measurement responses include `unlockedAchievements` that the UI ignores. Fix: mount the toaster pattern in those flows. Effort S.
- **E11.** Orphaned features: `/suggestions` (works, zero inbound links, not in nav), `trainers/my-trainer` and `trainers/requests` (zero inbound links), recipe favorites (linked only from a card). Fix: nav links. Effort S each.
- **E12.** Deep links lost on auth: `(dashboard)/layout.tsx:12-14` redirects to `/login` with no callbackUrl; login defaults to `/`. Fix: `?callbackUrl=`. Effort S.
- **E13.** Pricing `?plan=pro|lifetime` params silently dropped at register; no billing code exists — "Pro" features are ungated. Fix: remove tier CTAs or capture as waitlist. Effort S.
- **E14.** Logout confirmation modal is unnecessary ceremony; mobile bottom nav lacks a quick-log shortcut for the single highest-frequency action.

### Verified non-issues
JWT refresh is seamless (token cached, refetched 30s before expiry) — the 15-min JWT never forces re-auth.
